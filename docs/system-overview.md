# Image Upload Service – システム概要

## 1. 目的

本システムは、画像をアップロードして一時的に共有するための軽量なWebサービスである。

ユーザーはHTTP APIを通じて画像をアップロードできる。
サーバは画像を指定された最大サイズに収まるようリサイズし、WebP形式に変換してオブジェクトストレージへ保存する。

アップロード完了後、サーバは画像にアクセスするためのURLを返す。
このURLは一定時間のみ有効であり、期限を過ぎた画像はアクセスできなくなる。

本システムは以下の方針で設計されている。

* 運用コストをできるだけ低くする
* システム構成を単純にする
* 障害発生時の復旧を簡単にする

本サービスで扱う画像は重要データではない。
障害が発生した場合、画像が失われても問題ない前提とする。

---

## 2. システムの機能

本システムは以下の機能を提供する。

* HTTP API による画像アップロード
* 画像のリサイズ処理
* WebP形式への変換
* Cloudflare R2 への画像保存
* SQLite によるメタデータ管理
* 画像取得用URLの提供
* 画像の有効期限管理
* 期限切れ画像の削除

---

## 3. システム構成

本システムはできるだけシンプルな構成とする。

クライアントからのリクエストは Cloudflare を経由して VPS 上の Rust アプリケーションに到達する。

Rust アプリケーションがすべてのアプリケーションロジックを担当する。

構成概要:

Client (Browser)
↓
Cloudflare
↓
Rust Application (VPS)
├─ SQLite (metadata storage)
└─ Cloudflare R2 (image storage)

---

## 4. 各コンポーネント

### Cloudflare

役割:

* HTTPS終端
* サービスの公開入口
* DNS管理
* 基本的なトラフィック保護

CloudflareはリクエストをRustアプリケーションへ転送する。

---

### Rust Application

Rustサーバは以下の処理を担当する。

* フロントエンド静的ファイル（Vue build output）の配信
* `POST /api/images` による画像アップロード受付
* アップロード画像の検証
* EXIF orientation の適用
* 画像のリサイズ処理
* WebP形式への変換
* Cloudflare R2 への画像アップロード
* SQLite へのメタデータ保存
* `GET /i/{id}` による画像配信
* 画像の有効期限チェック

Rustアプリケーションは単一バイナリとして動作し、systemd により管理される。

---

### SQLite

SQLiteは画像メタデータの保存に使用する。

SQLiteを採用する理由:

* 運用コストが低い
* 別プロセスのデータベースサーバが不要
* 低トラフィック環境では十分な性能

SQLiteには以下の情報を保存する。

* `id`
* `object_key`
* `content_type`
* `width`
* `height`
* `size_bytes`
* `created_at`
* `expires_at`
* `deleted_at`

---

### Cloudflare R2

Cloudflare R2は画像データの保存先として使用する。

Rustアプリケーションは画像処理後のWebPファイルをR2へアップロードし、画像取得時にはR2から取得して返却する。

R2を使用する理由:

* 低コストなオブジェクトストレージ
* VPSのディスク容量を節約できる
* 将来的なスケールにも対応しやすい

---

## 5. 画像のライフサイクル

### アップロード

1. クライアントが画像をアップロードする
2. サーバがファイルを検証する
3. 画像をデコードする
4. EXIF orientation を適用する
5. 画像をリサイズする
6. WebP形式へ変換する
7. 画像をR2へアップロードする
8. メタデータをSQLiteへ保存する
9. クライアントへ画像URLを返す

---

### 画像取得

1. クライアントが `GET /i/{id}` にアクセスする
2. サーバがSQLiteからメタデータを取得する
3. `expires_at` を確認する
4. 期限切れの場合は `404 Not Found` を返す
5. 有効な場合はR2から画像を取得する
6. `image/webp` としてレスポンスを返す

---

### 有効期限

画像はアップロードから **12時間後に期限切れ**となる。

期限切れの画像は以下の状態となる。

* APIからアクセスできない
* R2には物理的に残っている可能性がある

---

### クリーンアップ処理

期限切れ画像の物理削除は **毎日12:00に実行する**。

削除処理は以下を行う。

1. `expires_at` が現在時刻より過去の画像を取得する
2. R2のオブジェクトを削除する
3. SQLiteの `deleted_at` を更新する

この処理は高頻度で実行する必要はない。

---

## 6. インフラ設計方針

本システムは意図的にインフラを簡素化している。

使用しない技術:

* Docker
* Kubernetes
* PostgreSQL
* 監視基盤

使用する技術:

* Rust単一バイナリ
* SQLite
* systemd
* cron

これにより運用の複雑さを最小限にする。

---

## 7. 障害時の方針

本サービスは高可用性を重視していない。

障害発生時の対応方針:

* サーバ再起動で復旧してよい
* 一部画像が失われても問題ない
* 手動対応でもよい

システムの単純さを優先する。

---

## 8. セキュリティ方針

本サービスでは最低限の安全対策のみを実施する。

実施する対策:

* 最大アップロードサイズ制限
* 画像フォーマット検証
* SVGの拒否
* EXIFメタデータ削除
* 画像デコード検証

本サービスは機密情報の保存用途ではない。
